## 1. 냅다 인프라에 던져진 프론트엔드 개발자: 단일 VM 배포 환경 구축기

_"분명 나는 Next.js 프론트엔드 개발잔데, 어쩌다 도커를 깎고 있는 걸까?"_

이번 글은 브라우저를 벗어나면 큰일 나는 줄 알았던 프론트엔드 개발자의 GCP 찍먹기입니다.

저희 회사에서는 gcp 기반으로 인프라를 관리하고 있는데요, 이전에도 가끔씩 인프라 작업에 도움을 주고 있었던 제가 이번에는 거의 혼자서(ai와 함께) 자사 홈페이지를 새롭게 배포하는 과정을 맡아 진행하게 되었습니다.

이번 프로젝트의 핵심은 자사 홈페이지를 `carbonsaurus-homepage` 프로젝트(Next.js 프론트 + NestJS 백엔드)를 GCP **Compute Engine VM 1대**에 Docker 기반으로 배포하고, GitHub Actions를 통한 CI/CD 자동화 파이프라인을 구축해 운영 안정성과 배포 효율성을 극대화하는 것이었습니다.

기존의 GKE 환경은 관리형 서비스로서 많은 이점을 제공했지만, 현재 서비스 규모와 운영 리소스를 고려했을 때 보다 직관적이고 비용 효율적인 VM 기반의 아키텍처가 적합하다고 판단했습니다. 단순히 서버를 구축하는 것에 그치지 않고, 다음과 같은 세 가지 지향점을 가지고 작업을 진행했습니다.

- **인프라의 코드화(IaC):** Terraform을 활용해 서버 생성부터 네트워크 설정까지 모든 인프라를 코드로 관리하여 재현 가능성을 확보하고자 했습니다.
- **배포 자동화(CI/CD):** GitHub Actions를 통해 코드 푸시부터 이미지 빌드, Artifact Registry 업로드, 실서버 배포까지 이어지는 파이프라인을 구축하여 휴먼 에러를 최소화하고자 했습니다.
- **프로덕션 수준의 보안과 최적화:** Nginx 리버스 프록시와 SSL 인증서 적용, 그리고 Docker 멀티스테이지 빌드를 통한 이미지 최적화 등 실무에서 요구되는 보안 및 성능 기준을 충족하는 것을 목표로 삼았습니다.

## 2. 작업의 세부 태스크들: 9단계의 여정

목표를 달성하기 위해 크게 애플리케이션의 컨테이너화, 클라우드 인프라 구축, 배포 자동화, 그리고 최종 환경 설정의 4가지 영역으로 나누어 총 9개의 세부 태스크를 수행했습니다.

### 영역 1: 컨테이너 최적화 및 오케스트레이션

- **태스크 1. 프론트(Next.js) · 백엔드(NestJS) Dockerfile 작성**: 모노레포 구조를 유지하면서 빌드 속도와 이미지 크기를 최적화하기 위한 멀티스테이지 빌드 환경을 구성했습니다.
- **태스크 2. `docker-compose.prod.yml` 작성**: DB, 백엔드, 프론트엔드, Nginx가 유기적으로 동작할 수 있도록 서비스 간 의존성과 헬스체크 설정을 포함한 오케스트레이션을 정의했습니다.

### 영역 2: CI/CD 파이프라인 구축

- **태스크 3. GitHub Actions CI/CD 워크플로우 작성**: 조직 내 공유 워크플로우를 재사용하여 효율적인 빌드 및 배포 자동화 체계를 만들었습니다. 변수(Variable)와 비밀값(Secret)을 분리 관리하는 법을 익히는 과정이었습니다.

### 영역 3: 인프라 프로비저닝 및 네트워크 설정

- **태스크 4. GCP Terraform 수정**: VM 생성, 방화벽 규칙, 고정 IP 예약 등 핵심 인프라를 Terraform 모듈로 관리할 수 있도록 코드를 작성하고 반영했습니다.
- **태스크 5. GCP Artifact Registry 저장소 생성**: 빌드된 이미지를 안전하게 보관할 저장소를 만들고, 서비스 계정(SA) 권한을 정교하게 제어했습니다.
- **태스크 6. DNS 설정 (AWS Route 53)**: 도메인 등록처와 DNS 관리처가 다른 복잡한 상황을 파악하고, A 레코드 등록을 통해 안정적인 도메인 연결을 완료했습니다.

### 영역 4: 운영 환경 고도화 및 데이터 이전

- **태스크 7. Nginx 리버스 프록시 및 HTTPS 설정:** Let's Encrypt를 통해 SSL 인증서를 발급받고, 보안 헤더와 리다이렉트 규칙을 적용하여 안전한 통신 환경을 구축했습니다.
- **태스크 8. PR 코드리뷰 반영:** 동료 개발자의 리뷰를 거쳐 빌드 스테이지 추가, 전용 DB 유저 사용 등 보안과 성능 면에서 한 단계 더 나아간 최적화를 진행했습니다.
- **태스크 9. MariaDB 데이터 임포트:** 기존에 운영 중이던 WordPress 데이터를 새 환경의 DB 컨테이너로 안전하게 이전하며 프로젝트를 마무리했습니다.

## 태스크 1. Dockerfile 작성

- **태스크 소개: 모노레포 환경의 컨테이너화**

가장 먼저 착수한 작업은 애플리케이션을 컨테이너화하는 것이었습니다. 우리 프로젝트는 `apps/frontend`와 `apps/backend`로 나누어진 모노레포(Monorepo) 구조를 가지고 있었습니다. 각 서비스는 독립적으로 빌드되어야 하지만, 루트에 있는 `package.json`이나 `pnpm-workspace.yaml` 같은 설정 파일을 공유해야 하는 특이점이 있었습니다. 따라서 빌드 컨텍스트를 프로젝트 루트(`.`)로 설정하고, 효율적인 빌드를 위해 멀티스테이지 빌드(Multi-stage Build) 방식을 도입했습니다.

- **직면한 어려움: Next.js 16 Turbopack과 경로의 늪**

작업은 순탄해 보였으나, PR 코드리뷰 내용을 반영하는 과정에서 예상치 못한 난관에 부딪혔습니다. 도커 이미지 크기 최소화를 위해 Next.js의 `output: 'standalone'` 설정을 추가했는데, 이를 적용하자마자 빌드 서버에서 에러가 발생하기 시작했습니다.

> Next.js 16 Turbopack이 워크스페이스 루트를 찾지 못합니다.

분명 로컬 환경에서는 문제가 없었는데, 도커 내부에서는 `apps/frontend/src/app` 경로에서 상위의 워크스페이스 루트를 인식하지 못하는 현상이었습니다. 처음에는 빌드 스테이지에 설정 파일들을 더 많이 복사(`COPY`)해 보기도 하고, `WORKDIR`을 서비스 디렉토리로 변경해 보기도 했지만 모두 실패했습니다. 백엔드는 문제없이 빌드되는데 프론트엔드만 실패하는 상황을 보며, 제가 도커의 빌드 환경과 모노레포의 경로 의존성을 충분히 이해하지 못하고 있다는 자책이 들기도 했습니다.

- **문제 해결과 기술적 이해**

이 문제를 해결하기 위해 시도한 디버깅 끝에, 결국 `next.config.ts`에서 Turbopack의 루트 경로를 명시적으로 재정의함으로써 실마리를 찾았습니다.

이 과정에서 해결 방법과 함께 정리한 기술적 개념들은 다음과 같습니다.

- **멀티스테이지 빌드(Multi-stage Build)를 통한 최적화:** 도커 이미지는 레이어가 쌓일수록 용량이 커집니다. 빌드 시에는 컴파일러나 각종 도구가 필요하지만, 실제 실행 시에는 결과물만 있으면 됩니다. 저는 `deps`(의존성 설치), `builder`(빌드 수행), `runner`(실행) 단계를 나누어 작성했습니다. 최종 `runner` 스테이지에서는 실행에 꼭 필요한 파일만 남김으로써, 보안을 강화하고 배포 시 네트워크 전송 비용을 줄이는 효과를 얻을 수 있었습니다.
- **Next.js Standalone 모드의 도입:** `output: 'standalone'` 설정은 Next.js가 프로덕션 실행에 필요한 최소한의 파일(노드 모듈의 일부와 소스 코드)만 별도의 폴더에 모아주는 기능입니다. 이 설정을 활성화하면 수백 메가바이트에 달하던 `node_modules` 전체를 복사할 필요 없이, `node server.js`만으로 서버를 띄울 수 있게 됩니다. 컨테이너 환경에서는 매우 효율적이지만, 빌드 시 전체 워크스페이스의 컨텍스트를 정확히 파악해야 한다는 전제 조건이 붙습니다.
- **Turbopack의 Workspace Root 인식 문제 해결:** Next.js 16에서 도입된 Turbopack은 성능을 위해 빌드 타임에 전체 프로젝트 구조를 분석합니다. 하지만 도커 스테이지 내부에서 하위 디렉토리로 이동하여 빌드를 수행할 때, Turbopack은 자신이 모노레포 내부에 있다는 사실을 망각하곤 합니다.

이를 해결하기 위해 `turbopack: { root: path.resolve(__dirname, '../..') }` 설정을 추가했습니다. `path.resolve`는 현재 디렉토리를 기준으로 절대 경로를 생성해 주는 Node.js API입니다. `../..`을 통해 도커 내부에서도 명확하게 모노레포의 루트 위치를 알려주었고, 그제야 Turbopack은 워크스페이스의 의존성 그래프를 정상적으로 그려내며 빌드에 성공할 수 있었습니다.

- **NestJS와 SWC 컴파일러의 조합:** 백엔드인 NestJS에서는 기존 `tsc` 대신 **SWC(Speedy Web Compiler)**를 사용했습니다. Rust 기반의 SWC는 컴파일 속도가 매우 빨라 빌드 시간을 단축하는 데 큰 도움이 되었습니다. 다만, NestJS 빌드 시 `.hbs` 같은 템플릿 파일은 자동으로 `dist` 폴더에 복사되지 않는 경우가 많습니다. 이를 위해 Dockerfile 내부에서 `cp` 명령어로 에셋들을 명시적으로 옮겨줌으로써 런타임에 파일 누락으로 인한 에러가 발생하지 않도록 조치했습니다.

첫 번째 태스크를 마치며, 단순히 "Dockerfile을 잘 쓴다"는 것이 문법을 아는 것에 그치지 않고, 프레임워크와 빌드 도구가 환경(컨테이너)과 어떻게 상호작용하는지를 깊이 파악해야 한다는 점을 배웠습니다.

## 태스크 2. `docker-compose.prod.yml` 작성

- **태스크 소개: 다중 컨테이너 오케스트레이션**

Dockerfile 작성을 통해 각각의 서비스를 이미지로 만드는 법을 익혔다면, 이제는 이 서비스들을 하나의 유기적인 시스템으로 엮는 과정이 필요했습니다. 이 태스크에서는 MariaDB, Backend(NestJS), Frontend(Next.js), Nginx라는 4개의 핵심 서비스를 정의하고, 프로덕션 환경에 적합한 설정값들을 `docker-compose.prod.yml` 파일에 담아내는 작업을 진행했습니다.

특히 단순한 실행을 넘어, Google Artifact Registry와 연동하기 위해 `${IMAGE_REGISTRY}/${서비스}:${IMAGE_TAG}`와 같은 동적 이미지 참조 패턴을 설계하고, 서비스 간의 시작 순서와 상태를 관리하는 데 집중했습니다.

- **직면한 어려움: 권한의 경계**

이 태스크에서는 데이터베이스 설정 과정에서 혼선이 있었습니다. 처음에는 `ROOT_PASSWORD`만 설정하면 모든 권한을 다 가질 수 있어 편할 것이라 생각했지만, 실제 운영 환경에서는 보안을 위해 관리자용 계정과 애플리케이션용 계정을 엄격히 분리해야 한다는 피드백을 받았습니다. "이 정보들을 내가 직접 정해도 되는 건가?" 혹은 "어디서 가져와야 하는 값인가?"와 같은 기본적인 설정 주체에 대한 고민부터, `IMAGE_TAG:-latest`와 같은 Bash 기본값 문법의 생소함까지, 인프라 초보자로서 넘어야 할 산이 꽤 많았습니다.

- **문제 해결과 기술적 이해: 안정적인 런타임 환경 구축**

이러한 문제들을 하나씩 풀어가며, 단순히 설정 파일을 작성하는 것을 넘어 서비스가 운영되는 기반 원리를 깊이 이해할 수 있었습니다.

- **최소 권한 원칙(Principle of Least Privilege)과 DB 보안:** 데이터베이스 설정 시 `MYSQL_ROOT_PASSWORD`와 `MYSQL_USER`를 분리하는 과정에서 보안의 기본 원칙을 배웠습니다. 애플리케이션 서비스가 루트 권한을 가질 경우, 혹여나 발생할 수 있는 보안 취약점이 데이터베이스 전체의 통제권 상실로 이어질 수 있습니다. 이를 방지하기 위해 전용 유저를 생성하고 필요한 권한만 부여하는 것이 운영의 정석임을 이해했습니다. 이 과정에서 `MYSQL_USER`, `MYSQL_PASSWORD`는 우리가 직접 정의하여 애플리케이션과 약속하는 "비밀번호 규칙"과 같다는 점을 명확히 인지하게 되었습니다.
- **서비스 헬스체크(Healthcheck)와 의존성 관리:** 단순히 `depends_on`만 사용하면 DB 컨테이너가 '시작'되자마자 백엔드 서버가 구동되려 시도합니다. 하지만 정작 DB가 '준비'되지 않아 연결 에러가 발생하는 경우가 잦았습니다. 이를 위해 `healthcheck` 설정을 추가하고, `depends_on: condition: service_healthy` 옵션을 적용했습니다. 이는 **시스템의 가용성(Availability)**을 보장하기 위한 핵심 기법으로, 각 서비스가 단순히 켜진 상태가 아니라 '요청을 처리할 준비가 된 상태'임을 확인하고 순차적으로 실행되도록 하여 전체 시스템의 부팅 안정성을 확보했습니다.
- **이미지 태그 전략과 가변성 통제:** `IMAGE_TAG:-latest`와 같은 문법은 환경 변수가 없을 때 기본값을 제공하는 Bash의 기능을 활용한 것입니다. 프로덕션 환경에서는 `latest` 태그 사용을 지양하고 명확한 버전이나 해시값을 사용하는 것이 권장되는데, 이러한 불변성(Immutability) 전략이 배포 시 발생할 수 있는 롤백 이슈나 버전 혼선을 어떻게 막아주는지 공부할 수 있는 계기가 되었습니다.

## 태스크 3. GitHub Actions CI/CD 워크플로우 작성

- **태스크 소개: 지속적 통합과 배포의 자동화**

Dockerfile과 `docker-compose`를 통해 실행 환경을 정의했다면, 이제는 코드가 업데이트될 때마다 자동으로 빌드와 배포가 이루어지는 파이프라인을 구축할 차례였습니다. 이번 태스크의 목표는 GitHub Actions를 활용하여 PR 생성 시 검증(CI)을 수행하고, 메인 브랜치 푸시 시 실제 서버에 반영(CD)하는 워크플로우를 작성하는 것이었습니다.

특히 조직 내에서 이미 사용 중인 공통 워크플로우를 재사용하여 표준을 따르면서도, 우리 프로젝트만의 특성(프론트/백엔드 병렬 빌드 등)을 반영하는 데 중점을 두었습니다.

- **워크플로우 재사용(Reusable Workflows)과 유지보수성:** 이 태스크에서는 기존 조직의 공유 워크플로우를 사용하기 위해 `uses:` 구문을 사용해 외부 워크플로우를 호출했습니다. GitHub Actions의 `workflow_call`과 `uses:` 기능은 소프트웨어 공학의 DRY(Don't Repeat Yourself) 원칙을 인프라 레벨에서 구현한 것입니다. 공통된 로직(예: GCP 로그인, 도커 이미지 빌드 및 푸시)을 하나의 중앙 저장소에 두고 각 프로젝트에서 호출해 쓰는 방식입니다. 이때 `secrets: inherit` 옵션을 사용하면 호출하는 쪽의 비밀값을 안전하게 전달할 수 있어, 보안과 편의성을 동시에 잡을 수 있다는 점을 다시 한번 확인할 수 있었습니다.

## 태스크 4. GCP Terraform 수정 (VM, 방화벽, 고정 IP)

- **태스크 소개: 코드로 구축하는 클라우드 인프라**

이제 자동화된 배포 파이프라인이 도착할 "목적지"를 만들 차례였습니다. 이번 태스크의 목표는 Terraform을 사용하여 GCP(Google Cloud Platform) 인프라를 프로비저닝하는 것이었습니다. 기존에 GKE 용도로 작성되어 있던 Terraform 코드에 Compute Engine VM 모듈을 추가하고, 서버가 생성되자마자 Docker와 gcloud SDK가 자동으로 설치되도록 `startup script`를 구성했습니다. 또한 외부 통신을 위한 고정 IP(Static IP) 예약과 서비스 포트(80, 443, 3000, 3004)를 허용하는 방화벽 규칙 설정을 포함했습니다.

- **IaC(Infrastructure as Code)와 상태 관리의 중요성:** Terraform은 선언적(Declarative) 언어입니다. "어떤 상태가 되어야 한다"를 코드로 기술하면 Terraform이 현재 상태와 비교하여 차이점(Diff)을 찾아내 반영합니다. 제가 경로를 착각하거나 OS 버전을 잘못 건드릴 경우 이 '상태(State)'를 오염시킬 수 있습니다. 인프라 코드는 애플리케이션 코드보다 훨씬 더 엄격하게 버전 관리가 되어야 하며, `terraform plan`을 통해 실행 전 변경 사항을 검토하는 습관이 왜 필수적인지 체감했습니다.
- **고정 IP(Static IP)와 고정된 엔드포인트의 필요성:** 일반적인 VM의 외부 IP는 재시작 시 변할 수 있는 임시(Ephemeral) IP입니다. 하지만 서비스 운영을 위해서는 도메인(DNS)에 연결할 변하지 않는 주소가 필요합니다. 이를 위해 `google_compute_address` 리소스를 사용하여 고정 IP를 먼저 예약했습니다. 이는 네트워크 통신의 **종단점(Endpoint)**을 고정하는 작업으로, 서버가 내부적으로 교체되더라도 외부에서 접근하는 주소는 유지되도록 하여 서비스의 연속성을 보장하는 핵심 설계임을 이해할 수 있었습니다.
- **리전(Region)과 존(Zone)의 계층적 구조:** 인프라를 구성하며 리전은 "지리적 위치(서울, 도쿄 등)", 존은 "해당 리전 내의 독립된 데이터 센터"라는 개념을 명확히 정리했습니다. 처음에는 리전(Region)과 존(Zone)의 개념이 모호해 "왜 Static IP는 리전 단위로 예약하면서 VM은 존 단위로 생성하는가?"에 대한 의문이 있었습니다. 고정 IP는 리전 전체에서 가용한 자원이기에 리전 단위로 예약하지만, 실제 물리적 서버인 VM은 특정 데이터 센터(존)에 종속됩니다. 이러한 계층 구조를 이해하고 나니, `google_compute_firewall` 규칙을 리전 내 전체 인스턴스에 적용할지, 특정 태그가 붙은 VM에만 적용할지에 대한 설계적 판단이 가능해졌습니다.

## 태스크 5. GCP Artifact Registry 저장소 생성

- **태스크 소개: 컨테이너 이미지의 중앙 저장소 구축**

인프라의 뼈대를 갖춘 후, 빌드된 도커 이미지들을 안전하게 보관하고 관리할 '중앙 창고'가 필요했습니다. 이번 태스크의 목표는 GCP의 Artifact Registry를 생성하고, 관련 서비스 계정(SA)에 적절한 권한을 부여하는 것이었습니다. 구체적으로는 `gcloud` 커맨드를 이용해 `asia-northeast3`(서울) 리전에 저장소를 생성하고, 배포 프로세스에서 이미지를 밀어넣고(Push) 가져올(Pull) 수 있도록 `terraform-sa` 계정에 IAM 권한을 설정하는 작업을 포함했습니다.

- **직면한 어려움: 이름의 일관성과 권한 부여의 대상**

이 태크스에서는 GCP 콘솔에 존재하는 여러 개의 서비스 계정 중 정확히 어떤 계정에 권한을 줘야 하는지 결정하는 것도 쉽지 않았습니다. "이미 있는 키를 써야 하나, 새로 만들어야 하나?" 하는 사소한 의문이 들기도 했습니다.

이러한 시행착오를 거치며 클라우드 자원을 관리하는 데 필요한 핵심 개념들을 하나씩 정립할 수 있었습니다.

- **역할 기반 액세스 제어(RBAC)와 최소 권한 원칙:** 서비스 계정에 권한을 부여하며 **IAM(Identity and Access Management)**의 핵심인 RBAC 개념을 익혔습니다. 무분별하게 '관리자(Owner)' 권한을 주는 대신, 이미지를 등록할 수 있는 `artifactregistry.writer`와 서버에서 이미지를 내려받을 수 있는 `artifactregistry.reader`를 구분하여 부여했습니다. 특히 `terraform-sa`라는 기계용 계정(Service Account)을 별도로 활용함으로써, 사람의 계정이 유출되더라도 인프라 제어권은 안전하게 분리될 수 있도록 설계했습니다.
- **서비스 계정 키 관리와 보안:** `gcloud iam service-accounts keys list`와 같은 명령어를 통해 기존에 발급된 키의 현황을 파악하는 법을 배웠습니다. 키를 무분별하게 새로 발급하기보다 기존 키의 유효성을 확인하고, GitHub Secrets와의 연결 구조를 파악함으로써 **기계 간 인증(Machine-to-Machine Authentication)**이 안전하게 이루어지는 흐름을 이해했습니다.

## 태스크 6. DNS 설정 (AWS Route 53)

- **태스크 소개: 가상 세계의 주소, 도메인 연결**

이미지를 보관할 창고와 이를 실행할 서버까지 준비되었으니, 이제 사용자들이 우리 서비스를 찾아올 수 있도록 '주소'를 부여할 차례였습니다. 이번 태스크의 목표는 GCP에서 예약한 고정 IP를 서비스 도메인에 연결하는 것이었습니다. 특히 기존에 운영 중인 서비스에 영향을 주지 않으면서 새로운 환경을 테스트하기 위해 `test.carbonsaurus.com`과 `api.carbonsaurus.com`이라는 서브도메인을 설정하고, 이를 AWS Route 53의 A 레코드(Address Record)로 등록하는 작업을 진행했습니다.

- **직면한 어려움: 도메인 관리처를 찾아 떠난 여정**

가장 먼저 맞닥뜨린 난관은 의외로 "우리 도메인이 어디서 관리되고 있는가?"를 찾는 것이었습니다. 당연히 국내 도메인 등록 업체인 '가비아'에서 관리하고 있을 것이라 짐작했지만, 가비아 관리 콘솔 어디에서도 해당 도메인을 찾을 수 없었습니다. 인프라 관리가 파편화되어 있을 때 발생할 수 있는 전형적인 문제였습니다.

또한, 기술적인 의구심도 생겼습니다. "프론트엔드와 백엔드의 IP 주소가 같아도 괜찮은가?" 하는 점이었습니다. 하나의 고정 IP에 두 개의 서로 다른 도메인(`test`, `api`)을 연결했을 때, 서버가 어떻게 이를 구분해서 적절한 컨테이너로 보내줄 수 있는지에 대한 확신이 부족했습니다.

- **문제 해결과 기술적 이해: DNS 시스템의 논리적 흐름**

도메인의 행방을 추적하고 레코드를 설정하는 과정에서, 인터넷의 전화번호부라 불리는 DNS(Domain Name System)의 작동 원리를 깊이 이해하게 되었습니다.

- **도메인 등록처(Registrar)와 DNS 호스팅(Name Server)의 분리:** 이번 작업을 통해 도메인을 '구매한 곳'과 '관리하는 곳'이 다를 수 있다는 점을 확실히 배웠습니다. `whois` 명령어와 `nslookup -type=NS`를 통해 확인해 보니, 도메인은 가비아에서 샀을지 몰라도 실제 네임서버는 AWS Route 53을 바라보고 있었습니다. 이는 대규모 인프라를 운영할 때 가용성과 관리 편의를 위해 DNS 관리 권한만 클라우드 서비스로 위임하는 전략임을 알게 되었습니다. 덕분에 '길을 잃었던' 도메인 설정의 주도권을 되찾을 수 있었습니다.
- **A 레코드(Address Record)와 IP 매핑:** DNS 설정의 핵심인 A 레코드는 도메인 이름을 IPv4 주소로 직접 연결해주는 역할을 합니다. 제가 예약했던 GCP의 고정 IP를 이 A 레코드에 등록함으로써, 전 세계 어디에서든 우리 도메인을 치면 GCP 서버로 찾아올 수 있는 이정표를 세웠습니다. 이때 TTL(Time To Live) 개념도 함께 익혔는데, 이는 DNS 레코드 정보가 인터넷상에 캐싱되는 시간을 의미합니다. 변경 사항이 빠르게 반영되어야 하는 설정 초기에는 TTL을 짧게 가져가는 것이 유리하다는 실무적인 팁도 얻었습니다.
- **서브도메인 전략과 가상 호스팅(Virtual Hosting):** 동일한 IP 주소를 여러 도메인이 공유할 수 있는 이유는 이후에 설정할 Nginx 같은 웹 서버가 HTTP 요청 헤더에 담긴 `Host` 필드를 보고 목적지를 구분하기 때문입니다. 이를 통해 물리적 자원(IP)을 아끼면서도 논리적으로 서비스(`test`, `api`)를 분리할 수 있다는 점을 이해했습니다. 특히 기존 운영 환경을 보호하기 위해 `test`라는 서브도메인을 활용한 것은, 실제 서비스 배포 시 위험을 최소화하는 안정적인 인프라 운영 전략의 일환임을 깨달았습니다.
- **고정 IP(Static IP)의 가치 재확인:** 만약 고정 IP를 사용하지 않았다면, 서버를 재시작할 때마다 AWS Route 53에 접속해 A 레코드를 매번 수정해야 했을 것입니다. 인프라 설계 단계에서 Static IP를 확보하는 것이 왜 DNS 설정의 선결 조건인지, 그리고 이것이 서비스의 영속성에 어떻게 기여하는지 체감할 수 있었습니다.

## 태스크 7. Nginx 리버스 프록시 + HTTPS(Let's Encrypt) 설정

- **태스크 소개: 안전하고 똑똑한 문지기 세우기**

인프라의 뼈대가 완성되고 도메인 주소까지 확보했으니, 이제 실질적으로 들어오는 요청을 안전하게 안내해 줄 '문지기'가 필요했습니다. 이번 태스크의 목표는 Nginx를 리버스 프록시(Reverse Proxy)로 설정하여 보안을 강화하고, Let's Encrypt를 통해 HTTPS 암호화 통신을 구현하는 것이었습니다.

단순히 통신만 연결하는 것이 아니라, HTTP로 들어온 요청을 HTTPS로 강제 리다이렉트하고, `test`와 `api`라는 두 서브도메인을 각각 프론트엔드와 백엔드 컨테이너로 정확히 전달하는 설계를 진행했습니다.

- **직면한 어려움: 자동화의 역설과 인증서의 복잡성**

작업을 시작하자마자 "Nginx가 정확히 무엇인지"부터 공부해야 했습니다. GKE 환경에서는 `cert-manager`나 Ingress가 자동으로 처리해 주던 영역이었기에, VM 환경에서 직접 바닥부터 설정하는 과정이 낯설게 다가왔습니다.

Let's Encrypt 인증서를 발급받는 과정도 만만치 않았습니다. 두 개의 도메인(`test`, `api`)을 하나의 통합 인증서로 발급받았더니, 인증서 파일들이 `test.carbonsaurus.com/`이라는 폴더 하나에 한꺼번에 저장되는 구조였습니다. 이를 인지하지 못해 API 서버 블록의 인증서 경로를 잘못 지정하여 한참을 헤매기도 했습니다. 여기에 더해 SSH 접속 중 갑작스러운 키 생성 메시지나 패스프레이즈 입력 요구는 운영 환경에 익숙하지 않은 저를 더욱 긴장하게 만들었습니다.

- **문제 해결과 기술적 이해: 보안과 효율의 접점 찾기**

수많은 에러 로그를 분석하고 설정을 다듬으며, 웹 서버가 통신을 처리하는 핵심 메커니즘을 배울 수 있었습니다.

- **리버스 프록시(Reverse Proxy)의 역할과 가상 호스팅:** Nginx는 클라이언트의 요청을 가장 앞에서 받아 내부의 적절한 서버로 전달해 주는 리버스 프록시 역할을 수행합니다. 이는 보안상의 이유로 내부 서버의 IP를 숨겨주는 방어막 역할을 할 뿐만 아니라, 하나의 포트(443)로 들어오는 요청을 `Host` 헤더에 따라 서로 다른 컨테이너(프론트/백엔드)로 분기해 주는 교차로 역할도 합니다. 이를 통해 한 대의 서버 안에서 여러 서비스를 독립적으로 운영할 수 있는 가상 호스팅의 원리를 이해했습니다.
- **TLS/SSL 인증서 발급과 SNI(Server Name Indication):** Let's Encrypt의 Certbot을 사용해 인증서를 발급받으며 HTTPS의 기초인 SSL/TLS 프로토콜을 공부했습니다. 특히 `standalone` 모드를 통해 별도의 웹 서버 구동 없이도 임시 서버를 띄워 인증서를 발급받는 과정을 익혔습니다. 여러 도메인이 하나의 인증서를 공유할 때 경로가 어떻게 생성되는지 확인하며, Nginx가 클라이언트가 요청한 도메인 이름을 보고 적절한 인증서를 내어주는 SNI 기술의 편리함을 느꼈습니다.
- **보안 헤더를 통한 방어 강화:** 단순히 암호화만 하는 것을 넘어, HSTS, X-Content-Type-Options, X-Frame-Options 같은 보안 헤더를 추가했습니다. 이는 브라우저 레벨에서 클릭재킹(Clickjacking)이나 MIME 스니핑 같은 공격을 원천 차단하는 장치입니다. 인프라 설정 한 줄이 서비스의 보안 수준을 얼마나 높일 수 있는지 배우며, '운영되는 코드'와 '안전하게 운영되는 코드'의 차이를 실감했습니다.

## 태스크 8. MariaDB 데이터 임포트 (수동)

- **태스크 소개: 데이터 마이그레이션과 프로젝트의 완성**

인프라 구축과 서비스 설정이 모두 완료되었으니, 이제 그 속을 채울 데이터를 옮길 차례였습니다. 이번 프로젝트의 마지막 관문은 기존에 운영 중이던 WordPress의 데이터베이스 덤프 파일(`my-wp-db.sql`)을 새로운 VM 환경의 MariaDB 컨테이너로 안전하게 임포트하는 것이었습니다.

단순히 파일을 옮기는 작업을 넘어, 원격 서버와 로컬 환경 사이의 데이터 전송 방식과 도커 컨테이너 내부로 데이터를 주입하는 스트림 구조를 이해해야 했던 작업이었습니다.

- **직면한 어려움: 도커의 빈 폴더 창조와 `rm -rf`의 공포**

마지막 단계라 가벼운 마음으로 시작했지만, 생각지도 못한 곳에서 시간을 뺏겼습니다. 도커에 덤프 파일을 마운트하려고 경로를 지정해 뒀더니, 파일이 아니라 냅다 동명의 '빈 폴더'를 창조해버리는 도커의 인성(?)을 마주했습니다.

파일을 덮어쓰려다 에러를 뱉어내는 터미널 앞에서 1차 당황. 결국 그 무시무시하다는 금지어 **`sudo rm -rf`**를 타이핑하며 2차로 손을 떨었습니다. (실수로 DB 날아가는 상상함 🤦‍♂️) GUI의 노예(?)였던 프론트엔드가 CLI의 매운맛을 제대로 본 순간이었습니다.

또한, `gcloud compute scp` 명령어를 실행하는 '위치'에 대해서도 혼란이 있었습니다. 서버 내부에 SSH로 접속한 상태에서 실행해야 하는지, 아니면 내 로컬 터미널에서 실행해야 하는지 헷갈려 명령어가 계속 실패하는 상황을 겪었습니다. 여기에 `docker exec`와 파이프(`|`, `<`)를 조합하여 데이터를 넣을 때 `sudo` 권한과 쿼팅 처리 문제까지 겹치면서, "데이터 한 줄 옮기는 게 생각보다 정교한 작업이구나"라는 것을 새삼 느꼈습니다.

- **문제 해결과 기술적 이해: 데이터 스트림과 파일 시스템의 원리**

파일 전송과 임포트 과정을 하나씩 디버깅하며, 리눅스와 도커가 파일을 다루는 근본적인 방식에 대해 학습할 수 있었습니다.

- **도커 볼륨 마운트와 호스트 파일 시스템의 상호작용:** 도커에서 호스트의 특정 파일 경로를 컨테이너와 연결(mount)할 때, 만약 호스트에 해당 경로의 파일이 존재하지 않는다면 도커는 기본적으로 이를 디렉토리로 간주하고 생성해버립니다. 이 현상을 겪으며 인프라 구성 시 파일이 생성될 순서와 존재 여부를 미리 체크하는 것이 얼마나 중요한지 배웠습니다. 특히 설정 파일이나 DB 덤프처럼 '파일' 단위의 마운트가 필요할 때는 더욱 주의가 필요함을 깨달았습니다.
- **표준 입력(STDIN) 리다이렉션과 `docker exec -i`:** 컨테이너 내부의 데이터베이스에 SQL 파일을 밀어 넣을 때 사용한 `docker exec -i mariadb mariadb -u... < dump.sql` 명령어는 리눅스의 표준 스트림(Standard Stream) 원리를 이용한 것입니다. `-i`(interactive) 옵션은 컨테이너의 표준 입력을 열어두어, 호스트의 파일 내용을 파이프를 통해 컨테이너 내부 프로세스로 흘려보낼 수 있게 해줍니다. 이 구조를 이해하고 나니, 굳이 대용량 파일을 컨테이너 내부로 일일이 복사(`docker cp`)하지 않고도 효율적으로 데이터를 주입할 수 있었습니다.
- **원격 전송(SCP)의 컨텍스트 이해:** `gcloud compute scp`는 로컬 장비의 파일을 클라우드 VM으로 전송하는 도구입니다. 이 명령어는 서버 내부가 아닌 **내 컴퓨터(Local)**에서 실행되어야 하며, 내부적으로는 SSH 프로토콜을 사용합니다.
